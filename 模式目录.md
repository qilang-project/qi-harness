# qi-harness 智能体模式目录（Agentic Patterns Catalog）

把 Google Cloud 归纳的 **12 个智能体（agentic）设计模式**逐一映射到 qi-harness 的对应模块，
每个模式给出：一句话定义 + 该模式在本框架里落地的关键点 + 一段**可直接编译运行**的最小
Qi 代码片段（100% 中文关键字，API 均取自各模块真实 `公开 函数` 签名）。

> 约定：下文所有片段假设已按需 `导入`。多数片段用环境变量取 provider（`QI_LLM_KEY` /
> `QI_LLM_BASE` / `QI_LLM_MODEL`），与 `examples/` 里既有示例一致。没有 LLM 时，纯本地
> 逻辑（如工具白名单、护栏、上下文估算）依然可跑。

## 1. Single-Agent —— 单智能体

**定义**：一个带系统提示与若干工具的智能体，接收任务、自主调用工具、给出答复。

**关键点**：`模型` 配置会话 → `创建代理` → `添加工具`（把 `工具` 注册进会话）→ `运行`（内部
自动 tool dispatch）。

```qi
导入 Harness::{ 模型配置, 大模型, 配置系统提示, 开启会话,
    创建代理, 添加工具, 运行, 关闭代理, 工具 };

函数 查天气(参数JSON: 字符串) : 字符串 {
    返回 "{\"city\":\"Beijing\",\"temp\":24,\"condition\":\"晴\"}";
}

函数 入口() {
    变量 配置: 模型配置 = 大模型("https://api.deepseek.com", "sk-...", "deepseek-chat");
    配置 = 配置系统提示(配置, "你是助手，可用工具查天气。");
    变量 代理值: 代理 = 创建代理("天气助手", 开启会话(配置));
    代理值 = 添加工具(代理值, 新建 工具 {
        名字: "查天气", 描述: "查询城市天气。",
        参数schema: "{\"type\":\"object\",\"properties\":{\"city\":{\"type\":\"string\"}}}",
        处理: 查天气,
    });
    打印行(运行(代理值, "北京今天天气怎样？"));
    关闭代理(代理值);
}
```

## 2. ReAct —— 推理-行动交替

**定义**：Reason（想）与 Act（调工具）交替，每次工具结果作为 observation 回喂，直到收敛。

**关键点**：`运行` 就是一个 ReAct 循环骨架——`工具调用 → 派发 → 回写工具结果(observation) →
继续对话`，用 `设置最大步数` 设上限防死循环，每步打 trace 可观测；配 `重试` 抗抖动。

```qi
导入 Harness::{ 创建代理, 设置最大步数, 添加工具, 运行, 工具, 启用 };
导入 Harness.追踪 作为 追踪;

追踪.启用即时打印();                 // 每轮 llm_call / tool_call / tool_result 打 JSON
变量 代理值: 代理 = 创建代理("排查助手", 会话);
代理值 = 设置最大步数(代理值, 6);    // ReAct 最多 6 步，超了返回上限提示，不无限循环
代理值 = 添加工具(代理值, 某工具);
变量 答复: 字符串 = 运行(代理值, "先查订单状态，再决定是否退款");
```

> 观察（observation）回喂由 `对话.回写工具结果(会话, 工具ID, 工具名, 结果)` 完成；
> `代理.运行` 已封装。抗瞬时错误见模式「横切关注点 · 错误」。

## 3. Sequential / Pipeline —— 顺序流水线

**定义**：把任务拆成固定阶段，一个阶段的输出是下一个阶段的输入，线性推进。

**关键点**：用 `图` 建有向图，`添加节点`(名字, 处理) + `添加边`(源, 目标) 串成链，`运行图`
从起点按边顺序推进，节点返回 `终点()` 结束。节点处理签名固定 `函数(字符串): 字符串`。

```qi
导入 Harness.图::{ 图, 创建图, 添加节点, 添加边, 运行图, 终点 };

函数 抽取(状态: 字符串) : 字符串 { 返回 "已抽取要点：" + 状态; }
函数 润色(状态: 字符串) : 字符串 { 返回 状态 + " → 已润色"; }
函数 收尾(状态: 字符串) : 字符串 { 返回 终点(); }   // 返回终点标记，运行图停止

变量 图值: 图 = 创建图();
图值 = 添加节点(图值, "抽取", 抽取);
图值 = 添加节点(图值, "润色", 润色);
图值 = 添加节点(图值, "收尾", 收尾);
图值 = 添加边(图值, "抽取", "润色");
图值 = 添加边(图值, "润色", "收尾");
变量 结果值: 字符串 = 运行图(图值, "抽取", "原始草稿", 10);
```

## 4. Parallel —— 并行扇出

**定义**：N 个相互独立的子问题同时问模型，几秒级的 LLM 调用并行摊平，结果按输入下标对齐收回。

**关键点**：`并行` 模块每个提示开**独立会话**（避免共享会话串话/竞态），`异步对话` 一次性发射、
再按序 `等待` 收割（fire-then-await）。用 `并行对话`（散参数）或 `并行映射`（模板套项目）。

```qi
导入 标准库.列表 作为 列表库;
导入 Harness.并行::{ 并行对话 };

变量 提示表: 整数 = 列表库::创建字符串列表();
列表库::添加字符串(提示表, "用一句话介绍东京");
列表库::添加字符串(提示表, "用一句话介绍巴黎");
列表库::添加字符串(提示表, "用一句话介绍开罗");
// 三路并发发出，结果[i] 对应 提示[i]
变量 结果表: 整数 = 并行对话("https://api.deepseek.com", "sk-...", "deepseek-chat", 提示表, 3);
```

## 5. Loop —— 目标循环

**定义**：让智能体在一个上限内自动迭代逼近目标，而不是人反复手敲 prompt。

**关键点**：`循环.运行循环` 是「制造者→校验者→反馈→再来」的目标驱动循环；`最大轮数` 是硬上限，
`自治级别`（L1 报告 / L2 协助 / L3 自动）调节校验者的升级阈值。见模式 6 的片段。

## 6. Review / Critique —— 生成-评审

**定义**：一个智能体产出，另一个智能体按硬约束验收；不达标就把反馈打回去重做。

**关键点**：`运行循环(干活代理, 校验代理, 目标, 最大轮数, 自治级别)`——干活代理走 `运行`（可用工具），
校验代理走 `简单问` 只读验收并输出严格 JSON 裁决（完成/升级/反馈）。`循环结果转文本` 打印全过程。

```qi
导入 Harness.循环::{ 运行循环, 循环结果, 循环结果转文本 };

变量 干活代理: 代理 = 创建代理("文案", 写作会话);
变量 校验代理: 代理 = 创建代理("验收", 审校会话);
变量 目标: 字符串 = "写一句不超过 14 字、含『Qi』、点出『100% 中文关键字』的标语";
变量 结果值: 循环结果 = 运行循环(干活代理, 校验代理, 目标, 5, 3);  // 5 轮上限, L3 自动
打印行(循环结果转文本(结果值));
```

## 7. Iterative Refinement —— 迭代精化

**定义**：同一产物一轮轮打磨，每轮基于上一轮结果继续改，逐步收敛到高质量。

**关键点**：仍是 `循环`——`运行循环` 内部把每轮的「成果 + 验收反馈」累加进跨轮 `状态` 字符串当
工作记忆，喂给下一轮制造者，从而「记得上轮改了什么、还差什么」。若干活代理 `启用记忆`，可用
`带记忆运行循环` 在收官时 `提炼经验` 落库，供跨任务复用。

```qi
导入 Harness.循环::{ 带记忆运行循环, 循环结果 };

变量 干活代理: 代理 = 启用记忆(创建代理("优化器", 会话), "./记忆.db");
// 每轮基于累积状态精化；收官把「怎么打有效」蒸馏成一条经验存库
变量 结果值: 循环结果 = 带记忆运行循环(干活代理, 校验代理, "把这段落改到既准确又简洁", 6, 3);
```

## 8. Coordinator —— 编排/worker（多专家协作）

**定义**：一个分诊/编排者把请求路由给合适的专家，专家之间也能横向转交（handoff）。

**关键点**：`多代理.团队` = 一组「名字 + 角色会话」的成员；`运行团队(团队, 起始名, 提示, 最大交接)`
从起始成员应答，成员若回一行 `转交给:成员名 转达内容` 就切换过去继续，直到普通作答或达上限。

```qi
导入 Harness.多代理::{ 团队, 创建团队, 添加成员, 运行团队 };

变量 团队值: 团队 = 创建团队();
团队值 = 添加成员(团队值, "分诊", 分诊会话);   // system: 判断该谁处理, 务必转交
团队值 = 添加成员(团队值, "退款", 退款会话);
团队值 = 添加成员(团队值, "技术", 技术会话);
变量 回复: 字符串 = 运行团队(团队值, "分诊", "我要申请退款", 3);  // 最多 3 次交接
```

## 9. Custom Logic —— 自定义控制流

**定义**：下一步走哪条分支由运行时状态决定，而非写死的线性顺序。

**关键点**：`图.添加条件边(源, 条件, 目标)`——`条件` 是 `函数(字符串): 整数` 谓词，读当前状态返回
1 才走这条边。同一源可挂多条条件边，实现 if/else、重试回边、循环等任意控制流。

```qi
导入 Harness.图::{ 图, 创建图, 添加节点, 添加边, 添加条件边, 运行图, 终点 };

函数 够好了(状态: 字符串) : 整数 { 如果 (字符串::包含(状态, "OK") == 1) { 返回 1; } 返回 0; }

变量 图值: 图 = 创建图();
图值 = 添加节点(图值, "生成", 生成处理);
图值 = 添加节点(图值, "结束", 结束处理);
图值 = 添加条件边(图值, "生成", 够好了, "结束");   // 达标才跳结束
图值 = 添加边(图值, "生成", "生成");                // 否则回到生成再来一轮
变量 结果值: 字符串 = 运行图(图值, "生成", "初始", 20);
```

## 10. Human-in-the-Loop —— 人在环

**定义**：在关键决策点暂停，把控制权交给人，人给出回复后从断点继续。

**关键点**：两条路——
(a) `图` 断点续跑：`运行图带检查点` 把每步状态落库，遇到需要人时中断；`是中断/中断问题` 取待答问题，
`恢复图(图, 库, 运行ID, 人工回复)` 带人的输入续跑（进程崩了也能按 运行ID 恢复）。
(b) `循环` 自治级别：L2 协助级下，校验者不确定/有风险就在裁决里 `升级=1`，把决定权交回人。

```qi
导入 Harness.图::{ 运行图带检查点, 恢复图, 是中断, 中断问题, 是完成 };

变量 结果值: 字符串 = 运行图带检查点(图值, 记忆库, "run-42", "审批节点", "初始状态", 30);
如果 (是中断(结果值) == 1) {
    打印行("需要人来定：" + 中断问题(结果值));
    变量 人答: 字符串 = 读一行();                  // 真实场景可跨进程再来
    结果值 = 恢复图(图值, 记忆库, "run-42", 人答);   // 带人工回复从断点续跑
}
```

## 11. Hierarchical —— 递归分解（本次新增 `递归分解` 模块）

**定义**：复杂任务先分解成子任务，子任务或直接完成或继续递归分解，最后自底向上汇总成最终答案。

**关键点**：`递归分解.递归求解(规划代理, 工人代理, 任务文本, 最大深度)`——规划代理用 `结构化运行`
产出结构化子任务列表，每个子任务递归下探一层（`最大深度-1`），叶子层交工人代理 `运行` 直接解，
再由规划代理 `简单问` 汇总。硬 `最大深度` 护栏防止无限展开导致成本失控；分解不可用时安全退化为
工人直接解，不崩。

```qi
导入 Harness.递归分解::{ 递归求解 };

变量 规划代理: 代理 = 创建代理("规划", 规划会话);  // system: 善于拆解与汇总
变量 工人代理: 代理 = 创建代理("工人", 工人会话);  // system: 认真完成单个子任务
变量 答案: 字符串 = 递归求解(规划代理, 工人代理,
    "为『奇语 Qi』写一段产品简介，涵盖定位、特色、人群", 3);  // 最大深度 3
```

## 12. Swarm —— 蜂群（本框架暂不实现）

**定义**：无中心的多智能体全网状协作——每个体都能与任意其他体来回辩论/投票，涌现出集体解。

**为何暂不做**：Swarm 是所有模式里 token/成本最高的一种——N 个体两两交互，消息量随对数增长，
且缺乏清晰的收敛与终止判据，容易「吵个没完」；工程可控性、可观测性、成本可预期性都差，属小众。
本框架已用 **`多代理`（有向 handoff，8）** 覆盖「专家协作」的高性价比部分，用 **`循环`（6/7）**
和 **`图`（3/9）** 覆盖「多轮收敛」，无需引入无约束的全网状辩论。真要蜂群，可在 `多代理` 上叠
`并行`（4）做多体并发 + 用 `循环` 的校验者做收敛裁决，但一般不建议。

## 13. Goal Setting & Monitoring —— 目标设定与监控（新增 `目标监控` 模块）

**定义**：给 agent 一个显式的目标 + 验收标准 + 里程碑清单，把「做到哪了」变成可量化、可断言的
确定性状态，而非全凭模型自述。

**关键点**：`目标监控` 核心全部 LLM-free——`进度`(已完成/总数)、`是否达成`、`目标报告`(✓/○ + 百分比)
都由里程碑标志位直接算；仅 `评估进度` 可选调一次 LLM 让模型判断达成情况。里程碑与完成标志用分隔串平行存储。

```qi
导入 Harness.目标监控::{ 创建目标, 添加里程碑, 标记完成, 进度, 是否达成, 目标报告 };

变量 目标值: 目标 = 创建目标("上线官网", "首页可访问且移动端适配");
目标值 = 添加里程碑(目标值, "写文案");
目标值 = 添加里程碑(目标值, "响应式布局");
目标值 = 添加里程碑(目标值, "部署生产");
目标值 = 标记完成(目标值, 0);            // 完成第 0 个里程碑
打印行(目标报告(目标值));                // 进度：1/3（33%）+ ✓/○ 清单
如果 (是否达成(目标值) == 1) { 打印行("目标达成"); }
```

## 14. Prioritization —— 优先级（新增 `优先级` 模块）

**定义**：给 agent 一个「先做什么」的调度器——任务带整数优先级，出队总取当前最紧急的。

**关键点**：`优先级` 核心全部 LLM-free——`入队/出队/队列大小/队列报告` 纯计算，`出队` 返回并移除最高
优先级任务（相等取先入队者，稳定）；仅 `评优先级` 可选调一次 LLM 给一批任务打分回填。任务与优先级用
分隔串平行存储。

```qi
导入 Harness.优先级::{ 创建队列, 入队, 出队, 队列报告 };

变量 队列值: 优先队列 = 创建队列();
队列值 = 入队(队列值, "写单元测试", 3);
队列值 = 入队(队列值, "修复线上 P0", 9);
队列值 = 入队(队列值, "更新文档", 1);
打印行(出队(队列值));      // 修复线上 P0（优先级最高 9）
打印行(出队(队列值));      // 写单元测试（3）
```

## 横切关注点（Cross-cutting concerns）

模式决定「怎么编排」，下列关注点决定「编排得可控」。它们与上面 12 个模式正交，随手叠加即可。

### 成本 → `预算`
token/成本硬上限。`创建预算(令牌上限, 成本上限分)` → 每轮 `记入(预算, 会话)` 累计真实用量 →
`超预算(预算)` 到顶即止，`预算报告(预算)` 出账单。

```qi
导入 Harness.预算::{ 创建预算, 记入, 超预算, 预算报告 };
变量 预算: 整数 = 创建预算(100000, 0);
记入(预算, 会话);
如果 (超预算(预算) == 1) { 打印行("预算耗尽，停止"); }
打印行(预算报告(预算));
```

### 可观测 → `追踪`（含 OTLP）
每次 LLM 调用 / 工具派发都打单行 JSON 事件，`开始跨度/结束跨度` 组成调用树；
`设置OTLP(端点)` 后每个完成的 span 直接 POST 到 OTLP collector。

```qi
导入 Harness.追踪::{ 设置OTLP, 新追踪, 开始跨度, 结束跨度 };
设置OTLP("http://127.0.0.1:4318");   // 转发到 collector 的 /v1/traces
新追踪();
变量 跨度: 整数 = 开始跨度("llm", "助手", "东京天气", 0);
结束跨度(跨度, "助手", "llm", "已回答", 1234, 0);   // tokens=1234, cost=0
```

### 上下文 → `上下文` + `工具上下文`
窗口管理：`估算令牌 / 需要压缩 / 滑窗裁剪 / 摘要压缩` 控制历史长度；
`工具上下文` 是工具处理函数间的旁路 KV（`设置工具上下文/取工具上下文`），避免把状态硬塞进 args。

```qi
导入 Harness.上下文::{ 需要压缩, 滑窗裁剪 };
导入 Harness.工具上下文::{ 设置工具上下文, 取工具上下文 };
如果 (需要压缩(会话, 3000) == 1) { 滑窗裁剪(会话, 6); }   // 只保留最近 6 条
设置工具上下文("用户ID", "u-42");                         // 工具内 取工具上下文("用户ID")
```

### 安全 → `护栏`
输入注入检测、PII 脱敏、输出字段校验、工具参数放行、一体化 `守卫运行`。

```qi
导入 Harness.护栏::{ 检测注入, 脱敏, 守卫运行 };
如果 (检测注入(用户输入) == 1) { 打印行("疑似提示注入，拒绝"); }
变量 干净: 字符串 = 脱敏(含隐私文本);        // 手机号/邮箱等打码
变量 安全回复: 字符串 = 守卫运行(代理值, 用户输入);  // 前置检测 + 后置守卫一把梭
```

### 治理 → 代理工具白名单（本次新增）
**每代理**精细权限：即便会话注册了 N 个工具，也能把某个代理限定到子集。
`设置工具白名单(代理, "工具A,工具B")` 之后，`运行` / `流式运行` 在 dispatch 前用 `工具是否放行`
判定，白名单外的工具**不执行**，直接回一条 `tool_not_allowed` 观测串给模型，并打 `tool_denied` trace。
白名单为空（默认）= 全放行，向后兼容，现有代理不受影响。

```qi
导入 Harness::{ 设置工具白名单, 工具是否放行 };
代理值 = 设置工具白名单(代理值, "查天气");   // 只准用查天气
// 之后 运行(代理值, ...) 里，模型若想调「删除文件」会被拒并收到错误观测，可自行改招
如果 (工具是否放行(代理值, "删除文件") == 0) { 打印行("删除文件 已被治理策略挡下"); }
```

### 错误 → `重试` + 步数上限
瞬时错误（429/网络抖动）自动退避重试见 `重试` 模块；逻辑层的失控由各模式的硬上限兜底
（`代理.最大步数`、`循环.最大轮数`、`图` 的 `最大步数`、`递归分解.最大深度`、`多代理` 的 `最大交接`）。

## 一句话选型

- 一个智能体能搞定 → **1 单智能体**；要调工具 → 天然就是 **2 ReAct**。
- 固定阶段流水线 → **3 图·顺序**；分支/回环由状态决定 → **9 图·条件边**。
- 独立子问题要快 → **4 并行**；要打磨到达标 → **5/6/7 循环**。
- 多专家分工 → **8 多代理 handoff**；关键点要人拍板 → **10 人在环**。
- 大任务要分治 → **11 递归分解**。**12 蜂群** 一般不用。
- 任何模式都别忘了叠 **成本/可观测/上下文/安全/治理/错误** 六个横切关注点。
