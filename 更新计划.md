# qi-harness 更新计划

本文定义 qi-harness 从当前 `0.1.0` 向更稳定 Agent Runtime 演进的实施顺序。计划参考 Pi Agent 在分层、生命周期事件、工具执行管道和持久化会话方面的实践，但不把 qi-harness 改造成 Coding Agent，也不删除现有的图工作流、RAG、MCP、评估、OTLP、多代理和安全工具能力。

## 目标

本轮更新优先解决四类问题：

1. 已有的重试、预算、护栏和上下文管理没有进入默认 Agent 执行链，需要从“可手动组合”升级为“可配置且默认一致”。
2. 当前 trace 适合记录结果，但缺少稳定的运行时生命周期事件，UI、服务、存储和扩展难以在不修改核心循环的情况下接入。
3. 工具接口简单易用，但缺少统一的参数验证、执行前后钩子、取消、进度更新和错误结果协议。
4. 图工作流已经支持断点续跑，普通对话仍主要依赖进程内会话句柄，缺少持久化、恢复和分支能力。

更新完成后，qi-harness 应同时具备：

- 小型 Agent 仍可用少量代码启动。
- 生产服务可以配置重试、预算、权限和上下文策略。
- CLI、Web、日志、持久化和第三方扩展通过统一事件接入。
- 工具失败不会破坏 Agent Loop，并能给模型提供可恢复的结构化反馈。
- 对话和工作流都可以恢复，但两者保持独立的数据模型。
- 现有公开 API 在有明确迁移方案前不做无必要破坏。

## 设计原则

### 保留 qi-harness 的定位

qi-harness 继续定位为通用业务 Agent 框架，不转向只服务编码场景。以下能力继续作为核心差异化能力维护：

- MCP 客户端和服务端
- 图工作流、检查点和 Human-in-the-loop
- RAG、向量记忆和混合检索
- 评估、baseline、回归门禁和 tape replay
- OTLP、运行报告和真实 token 用量
- 多代理 handoff、递归分解和并行调用
- 文件沙箱、工具白名单和应用级护栏

### 先修执行语义，再拆包

当前最重要的问题不是目录数量，而是默认执行链是否可靠。先完成重试、预算、事件、工具结果和会话协议，再根据稳定边界拆分模块。避免在协议尚未确定时提前制造多个包。

### 默认简单，高级能力显式配置

`创建代理` 和 `运行` 保持低门槛。重试、预算、事件、会话存储等能力通过配置附加，不要求简单示例一次性理解全部概念。

### 错误是模型可见的结果

工具不存在、参数无效、权限拒绝、执行异常和超时都应转成统一工具结果回写给模型。只有框架自身状态损坏、协议不一致等不可恢复错误才终止运行。

### 应用控制不是系统隔离

文件沙箱、白名单和护栏继续提供最小权限，但文档必须明确：它们不是容器、虚拟机或系统调用沙箱。执行不可信命令和第三方 MCP server 时仍应使用 OS 级隔离。

## 非目标

本轮不计划：

- 复制 Pi 的完整 TUI 或主题系统。
- 把所有模块一次性拆成多个仓库或包。
- 为保持旧内部实现而增加长期兼容层。
- 承诺任意外部副作用的严格 exactly-once。
- 用复杂语义策略替代现有简单护栏，并把它宣传为完整安全边界。
- 删除现有高级功能以追求最小代码量。

## 当前进度

截至 P2-P6 首轮实现，以下项目已经完成：

- `配置超时` 已接入 runtime 的 `timeout_secs`，本地慢服务测试证明 1 秒超时实际生效。
- `简单问`、`运行`、工具续调和结构化输出已统一接入非流式智能重试。
- Agent 可配置重试策略，并可附加跨会话 token/成本预算账本。
- 每次成功的非流式 provider 请求会自动累计真实 usage；达到上限后下一次请求在 provider 前被阻止。
- Agent 和结构化输出遇到 provider、预算、熔断或重试耗尽错误时会立即返回，不再解析为助手消息或继续 Schema 重试。
- 预算阻止和失败调用不再输出上一次请求的旧 usage。
- `/chat` 在注册服务工具时进入工具循环；无服务工具时保持纯问答；`/json` 保持无工具的结构化输出。
- 离线测试已有统一入口 `./run-offline-tests.sh`，断言失败会返回非零退出码。
- `tests/m1_reliability/run.sh` 使用随机高位本地端口，确定性覆盖超时、自动预算、工具首调和工具续调。
- 新增 lifecycle v1 envelope 与同步订阅总线；非流式 Agent 路径发出 Agent、turn、LLM 和工具 start/end 事件。
- trace 与报告提供显式安装的 lifecycle adapter；没有消费者时 Agent 业务结果不依赖适配器。
- 工具派发验证参数和 before hook 输出为 JSON 对象，支持 before/after hook、标准错误 helper 及执行模式。
- 一批工具调用仅在全部为已放行的本地并发工具时并发，其他情况整批串行，结果按原始顺序回写。
- 每个新 Agent 默认拥有独立重试资源；多个 Agent 可通过 `共享重试资源` 显式共享熔断和限流。
- 非流式运行建立并清理显式 run context，工具上下文委托给当前 run 的 request-scoped 键值表。
- SQLite 会话存储采用 append-only entry 树和 current leaf，支持非破坏分支、Agent 自动记录与显式 runtime 恢复。
- 会话支持确定性的版本 1 JSON 导出，以及校验 parent/leaf/ID 冲突后的单事务导入。
- HTTP 可通过 `配置服务持久会话` 使用不透明 UUID、SQLite 恢复和 owner 相等校验；旧 runtime-handle 协议仅在未配置持久模式时保留。
- 最小 CLI 支持 provider 环境诊断和 SQLite 会话 `list/show/path/export/import/branch`；`test` 显示统一门禁入口。
- `./run-offline-tests.sh` 已纳入事件、适配器、工具管道/调度、run context、会话导入导出与 Agent 集成、
  CLI、服务持久化、文件沙箱/报告/检索隔离和 M1 reliability。
- GitHub Actions 已在 Ubuntu 22.04 与 macOS 14 上安装固定 Qi 版本并运行统一离线门禁。

当前实现仍不是完整生产 Runtime：

- 流式和异步调用尚未统一接入自动重试、usage/预算、canonical lifecycle 与会话记录。
- lifecycle 已有显式 bus，但 Agent 固定发布到默认 bus；legacy trace sink、默认报告、服务配置和旧检索兼容 API
  仍有进程级默认状态。报告、文件沙箱和检索配置已有显式隔离路径并进入门禁。
- 工具管道只验证 JSON 对象形状，不执行完整 JSON Schema；取消、进度、工具超时和 panic 隔离未实现。
- SQLite 存储已有 `:memory:` 模式，但不是独立内存 backend；尚无 JSONL backend、自动配置恢复、通用租户策略或加密。
- HTTP 持久模式仍使用模块级配置/工具/DB，owner 只是字符串相等校验；未知/owner 不匹配统一 404。
  已有同进程 SQLite 请求锁和 409 冲突，但缺少真正认证、会话关闭、跨实例并发策略和流式 lifecycle 输出。
- HTTP 未启用持久模式时仍接收进程内 runtime `会话号`，该兼容协议不适合不可信客户端。
- CLI 不是 Agent runner；当前覆盖诊断、质量门禁说明、会话查看、导入导出和移动 leaf 分支。

## P1：统一运行配置

**状态：部分完成。** 非流式模型调用已统一，Agent 上已有重试策略、独立重试资源、预算和最大步数；
尚未形成单一 `运行配置` 类型，上下文压缩与护栏也未自动进入统一链，流式路径仍独立。

### 目标

把散落的可靠性能力组合进一个明确的 Agent 运行配置，同时保留简单默认值。

### 建议接口

新增逻辑上的 `运行配置`，至少包含：

- 最大工具步数
- LLM 重试次数、初始退避和最大退避
- 请求超时
- token 预算和成本预算
- 上下文压缩阈值与策略
- 是否启用输入、工具参数和输出护栏
- 是否记录 trace、usage 和报告
- 取消信号或运行取消句柄

优先让配置附着于 `代理` 或单次 `运行`，避免继续增加进程级全局开关。

### 执行语义

- 每次模型调用都经过统一调用函数，而不是由各入口分别直接调用 runtime。
- 重试只覆盖可重试错误，不重试鉴权失败、无效请求和明确的策略拒绝。
- 每次模型响应后自动读取真实 usage 并累计预算。
- 达到预算时停止下一次模型调用，返回明确的 `budget_exceeded` 结果并发出事件。
- 超时和取消需要区分，便于上层决定是否恢复或重试。
- 上下文压缩在模型调用前执行，并产生可观测事件。

### 验收标准

- `运行`、`流式运行` 和结构化输出共用同一可靠性调用路径。
- 模拟两次瞬时失败、第三次成功时，只执行三次调用并保留完整事件顺序。
- 预算达到上限后不再发出新的 provider 请求。
- 不可重试错误不会进入退避循环。
- 超时配置有 runtime 级测试证明实际生效。

## P2：生命周期事件

**状态：部分完成。** 已有 `生命周期事件` v1、显式生命周期事件总线、按总线订阅/发布 API，以及
兼容的默认总线 API。交付是进程内同步、按注册顺序；`隔离失败` 只能
隔离回调返回 `0`，不能捕获 panic。trace/report 需调用 `安装生命周期追踪适配器` /
`安装生命周期报告适配器` 显式订阅。

当前非流式 Agent 发出 `agent_start/end`、`turn_start/end`、`llm_start/end`、`tool_start/end`。
`llm_delta`、retry/budget/context/checkpoint 事件和流式 lifecycle 尚未完成。事件载荷目前多为空对象，
报告无法仅靠现有 Agent lifecycle 获得完整 token/cost/detail。Agent 仍固定发到默认 bus，尚不能把
自定义 bus 句柄附加到单个 Agent。

### 目标

在现有 trace 之外建立正式的运行时事件协议。事件用于扩展和协作，trace 用于观测和导出；trace 可以由事件生成，但二者不应混为同一个概念。

### 最小事件集

- `agent_start` / `agent_end`
- `turn_start` / `turn_end`
- `llm_start` / `llm_delta` / `llm_end`
- `tool_start` / `tool_update` / `tool_end`
- `retry_start` / `retry_end`
- `budget_update` / `budget_exceeded`
- `context_compaction_start` / `context_compaction_end`
- `checkpoint_saved` / `run_resumed`
- `error`

每个事件至少携带：

- 事件类型
- trace ID 和可选 parent span ID
- Agent 名称
- run ID
- turn 或 step 编号
- 时间戳
- 事件特有数据

### 订阅语义

- 生命周期开始和结束事件按顺序同步交付，保证消费者看到一致状态。
- 高频 delta 和工具进度允许批量或异步交付，但必须保证结束事件在全部已接受更新之后。
- 内部可信订阅者的错误默认向上返回；第三方扩展订阅者应有隔离选项。
- 订阅者可以读取事件，第一版不允许任意修改核心状态。

### 与现有 trace 的关系

- 现有 JSONL、文件 sink、报告和 OTLP 通过显式 adapter 消费事件；是否以后默认安装需结合重复输出与兼容性决定。
- 保持当前 trace 输出字段兼容，除非进入明确的主版本升级。
- 业务应用可以订阅事件并转发 SSE、WebSocket 或消息队列，无需修改 `代理.运行`。

### 验收标准

- 一次无工具调用产生完整且配对的 Agent、turn 和 LLM 事件。
- 一次多工具调用产生有序的 start/end，进度不会出现在对应 end 之后。
- trace、报告和 OTLP 可以仅通过订阅事件生成。
- 移除所有事件消费者后，Agent 的业务结果不变。

## P3：工具执行管道

**状态：部分完成。** 当前管道为：查找 → JSON 对象验证 → before hook → 本地/MCP 执行 →
`[harness]` 错误规范化 → after hook → 回写。`工具成功` / `工具错误` 提供标准 JSON helper，
但旧处理器结果不会自动包装。`设置执行模式` 使用 `0=并发、1=串行`；本地默认并发，MCP 固定串行。

尚缺完整 JSON Schema 验证、权限策略统一 hook、取消、进度、工具超时、异常捕获和可终止运行的
标准结果字段。before/after hook 是无状态函数，并且 hook/处理器 panic 仍可能终止进程。

### 目标

保留当前简单工具定义，同时在运行时增加统一的受控执行管道。

### 管道顺序

```text
查找工具
→ 参数预处理
→ JSON Schema 验证
→ 权限和策略检查
→ before hook
→ 执行工具
→ after hook
→ 结果标准化
→ 回写模型
```

### 标准工具结果

工具不再只依赖任意字符串表达成功或失败。新增标准结果概念，至少包含：

- 是否成功
- 文本或 JSON 内容
- 错误代码
- 可供模型修正的具体错误说明
- 可选展示元数据
- 可选是否终止当前 Agent 运行

为降低迁移成本，现有 `函数(字符串):字符串` 处理器可以先由适配器包装成成功结果；新工具优先使用标准结果接口。

### 调度语义

- 工具声明执行模式：`并行` 或 `串行`。
- 未声明时采用保守默认值；只读工具可以显式并行，写操作默认串行。
- MCP 连接按 transport 能力决定是否并发，不再把所有 MCP 永久等同为串行。
- 并发执行后，回写模型的 ToolResult 顺序与原始 ToolCall 顺序一致。
- 工具可接收取消信号并上报进度。

### 错误处理

- 未知工具、Schema 错误、白名单拒绝、超时和执行异常统一转换为工具错误结果。
- 错误描述应包含“哪里错、为什么错、如何修正”所需的信息，避免只返回“执行失败”。
- hook 和扩展异常需要有明确隔离策略，不能静默丢失。

### 验收标准

- 无效参数不会进入工具处理函数。
- 被白名单拒绝的工具不会产生副作用，并向模型返回标准错误结果。
- 工具抛出异常时 Agent Loop 仍可继续，由模型决定下一步。
- 两个只读工具可并行，两个写工具按声明串行。
- 取消运行后，支持取消的工具能停止并产生 `cancelled` 结果。

## P4：持久化会话

**状态：部分完成。** `会话存储` 已实现 append-only entry、current leaf、历史加载、
root-to-leaf 路径、非破坏分支、版本化导入导出、消息/工具/错误/配置/摘要 entry helper，以及
`恢复会话到运行时`。Agent 可用 `附加会话存储` 自动记录非流式执行，并用 `恢复持久会话` 显式恢复。

当前只有 SQLite backend；`打开内存会话存储()` 使用 SQLite `:memory:`，关闭唯一句柄后数据消失。
恢复重放消息、原始助手响应和工具结果，不自动恢复模型/配置/摘要语义；没有独立内存/JSONL backend、
自动压缩视图、通用租户策略或加密。导入会验证版本、parent 顺序、leaf 引用与 ID 碰撞并在单事务中
写入，相关测试和 Agent 恢复/记录集成测试已进入统一门禁。

### 目标

为普通 Agent 对话增加独立于图检查点的持久化模型。图检查点保存工作流执行位置；会话存储保存用户、模型、工具和配置变化形成的对话历史。

### 数据模型

采用 append-only entry 模型，每条记录至少包含：

- entry ID
- session ID
- parent entry ID
- entry 类型
- 时间戳
- payload

第一批 entry 类型：

- 用户消息
- 助手消息
- 工具结果
- 模型或运行配置变更
- 上下文压缩摘要
- 分支摘要
- 会话元数据

### 存储接口

定义可替换的会话存储接口，首批实现：

- SQLite：默认持久化实现，复用项目现有数据库能力。
- 内存：测试和临时运行。
- JSONL：可选的人类可读导出或轻量本地实现。

不要让具体文件格式进入 Agent Loop。Agent Loop 只依赖会话读写接口和当前分支视图。

### 核心操作

- 新建会话
- 追加 entry
- 加载会话
- 从当前 leaf 构建线性模型上下文
- 回退到指定 entry
- 从历史节点创建新分支
- 压缩旧上下文但不删除历史 entry
- 导出和导入会话

### 与 runtime 会话的关系

- 持久化会话是事实来源，runtime session 是执行缓存。
- 恢复时由持久化 entry 重建 runtime 历史，而不是保存并信任旧的整数句柄。
- 服务端不再允许客户端直接提交无归属校验的 runtime session handle。

### 验收标准

- 进程退出后可以按 session ID 恢复对话并继续调用工具。
- 从历史节点创建分支不会删除原分支。
- 压缩后当前上下文变短，但原始历史仍可导出和回看。
- 模型或配置切换记录在分支上，回退后自动恢复当时配置。
- 不同用户或租户不能通过猜测 ID 访问其他会话。

## P5：状态隔离与并发

**状态：部分完成。** 已有显式 run context 句柄，保存 run ID、轮次/步骤、trace metadata 和工具键值；
非流式 Agent 会建立并清理 context。重试熔断/限流默认按 Agent 分配资源，也支持显式共享。

这不等于全部状态已经实例隔离。报告已支持独立句柄与按句柄 lifecycle adapter，但 lifecycle
subscribers、legacy trace 当前 ID/sink、默认报告、
embedding 默认兼容配置和 HTTP 服务配置仍需继续清理。文件沙箱、报告和检索配置已有显式句柄隔离并
进入门禁，但 legacy 默认 API 仍存在。当前上下文栈与句柄表也是进程内实现。

### 目标

消除影响多 Agent 和多租户服务的可变全局配置，将状态归属到 Agent、run 或 service instance。

### 优先处理

- 追踪当前 trace 和 sink
- 文件工具当前沙箱
- 默认报告指针与 legacy 无句柄报告 API
- 重试熔断器和限流器
- embedding 和检索配置
- HTTP 服务配置和工具集合
- 工具上下文

### 状态归属规则

- Agent 长期配置归 Agent。
- 单次运行数据归 run context。
- request-scoped 数据归工具上下文，并在运行结束后清理。
- provider 级限流和熔断归明确的共享资源对象，不使用匿名进程全局单例。
- 只读常量可以继续全局共享。

### 验收标准

- 同一进程中两个 Agent 使用不同文件沙箱时互不覆盖。
- 并行运行的 trace ID、报告和工具上下文不串线。
- 不同 provider 可以配置独立限流和熔断状态。
- 并发测试可重复运行，无数据竞争和偶发失败。

## P6：逻辑分层与产品入口

**状态：部分完成。** 新增 `事件`、`运行上下文`、`会话存储`、`命令行` 模块；前三者由 `Harness.qi`
公开导出，`命令行` 仍按子模块导入；没有进行物理拆包。最小 CLI 已提供 `check`、`test` 和
`session list/show/path/export/import/branch`。`check` 不联网，`test` 只打印门禁命令；CLI 尚未达到
计划中的 Agent 运行、恢复后继续执行、trace 导出和 MCP/embedding 深度诊断。

HTTP 适配器当前语义保持兼容：`/chat` 有工具时走 Agent loop，无工具时纯问答；`/json` 无工具。
调用 `配置服务持久会话` 后使用不透明 UUID，未知 ID、owner 不匹配或映射缺失统一返回 404；每个请求
新建 runtime、恢复 SQLite、执行并关闭。未配置时仍保留旧 `会话号` 协议。owner 不是认证机制，
服务状态仍为模块全局。已有请求/提示应用层大小上限和同进程会话锁，但 `/health` 仅是 liveness，且
缺少关闭路由、传输层限流、跨实例并发策略和 SSE/WebSocket lifecycle，因此 HTTP 验收标准只完成了
持久 ID、基础归属隐藏和同进程冲突保护，尚未完成生产服务要求。

### 目标

在核心协议稳定后整理模块边界，并提供最小但完整的产品入口。

### 建议逻辑层

```text
模型与消息层
  provider 配置、消息、流式事件、usage

Agent Runtime 层
  Agent Loop、工具协议、事件、运行配置、会话接口

Harness 能力层
  工作流、RAG、记忆、评估、预算、护栏、多代理、MCP

适配层
  CLI、HTTP、MCP server、报告和 OTLP
```

第一步可以只调整模块依赖和公开入口，不强制立刻拆成多个独立包。只有当底层能够在不导入上层模块的情况下独立编译和测试，才考虑物理拆包。

### 最小 CLI

提供一个面向开发和诊断的 CLI，而不是直接复制完整 Coding Agent：

- 运行指定 Qi Agent 程序
- 列出和恢复会话
- 导出 trace 或会话
- 运行离线评估
- 检查 provider、MCP 和 embedding 配置

### HTTP 适配器

- 使用持久化 session ID，不暴露 runtime handle。
- 支持流式事件输出。
- 提供认证、租户归属校验和会话关闭。
- `/chat` 与 `/json` 明确是否启用工具循环。
- 服务实例不依赖进程级全局配置。

### 验收标准

- Agent Runtime 层可以独立完成无 UI 的工具循环测试。
- 工作流和 RAG 只依赖 Runtime 的公开协议，不反向污染模型层。
- CLI 和 HTTP 都通过事件和会话接口接入，不复制 Agent Loop。
- HTTP 会话具备归属校验和资源清理策略。

## 成功指标

- 简单 Agent 的入门代码不比当前明显复杂。
- 默认 `运行` 不再遗漏已声明启用的重试、预算和超时策略。
- 新增 UI、日志或服务转发不需要修改 Agent Loop。
- 工具参数错误和执行异常不会导致整个循环意外崩溃。
- 会话可以跨进程恢复，且支持非破坏性分支。
- 两个不同配置的 Agent 可以安全地在同一进程并发运行。
- README、Skill、示例和实现之间不存在已知能力描述偏差。
